    [Rust] 전체 흐름 정리 — 한 글 작성의 여정

    00~12편을 거치며 board-api의 소스·템플릿·테스트를 한 줄씩 따라왔습니다. 이 글은 전체 지도로 복습하고, “글 하나가 저장되기까지” 한 요청의 여정을 다시 연결합니다.

    독자별 읽기 가이드¶

    배경 추천
    Rust를 전혀 모름 00 → 01~03을 천천히 → cargo run으로 00 실습 → 04~07 → 나머지
    Rust는 알지만 웹 프레임워크가 처음 00, 04~12를 중심으로, 01~03은 참고
    백엔드 시니어 00·07·08·13 + 소스 직접 대조. 각 편 「심화」 절 위주

    막히면 컴파일러 에러 메시지를 그대로 검색하는 것이 Rust 학습에서 가장 빠른 길입니다. cargo check 출력은 “어느 파일·어느 타입이 안 맞는지”를 정확히 알려 줍니다.

    레이어 파일 책임

    REST vs Web — 같은 심장, 다른 피부¶

    REST (/api/posts) Web (/posts, /)
    입력 Json<CreatePostRequest> Form<PostForm> → DTO
    출력 Json<PostResponse> Askama Html
    에러 JSON { "error": ... } HTML 에러 페이지 또는 폼 재표시
    삭제 DELETE POST .../delete

    07·10·11편에서 강조한 대로 services/post.rs는 한 벌입니다.

    한 글 작성의 여정 (HTML)¶

    브라우저가 새 글 폼을 제출할 때:

    1. POST /posts — routes/web.rs create_post
    2. Form<PostForm> → into_create_request()
    3. services::create_post — validate_create, ActiveModel::insert
    4. SeaORM → SQLite posts 행 추가
    5. 성공 시 Redirect /posts/{id}
    6. GET /posts/{id} — get_post → DetailTemplate 렌더

    JSON API는 3~4까지 동일하고, 5~6 대신 JSON PostResponse를 반환합니다.

    한 글 작성의 여정 (REST)¶

    POST /api/posts → routes/post.rs → services → DB
← 200 JSON PostResponse

    tests/api.rs가 이 경로를 oneshot으로 검증합니다.

    자주 묻는 질문 (FAQ)¶

    Q. Rust 없이 Spring으로만 하면 안 되나요?
    A. 물론입니다. 이 시리즈는 Rust + Axum 생태계 읽기가 목표입니다.

    Q. 컴파일이 너무 느려요.
    A. 첫 빌드는 의존성 전체 컴파일입니다. cargo check로 개발하고, cargo build --release는 배포 전에만 쓰면 됩니다.

    Q. ?가 뭔지 아직 헷갈려요.
    A. “실패하면 여기서 함수 끝내고 Err를 위로 넘긴다”로 기억하세요. 03·08편 예제를 return Err(...)로 풀어 써 보면 같습니다.

    마치며¶

    이 시리즈의 목표는 Axum + SeaORM 게시판 예제의 모든 파일을 스스로 설명할 수 있는 것이었습니다. 새 코드를 읽을 때는 항상:

    1. URL이 어느 route인지
    2. service에 무엇을 맡기는지
